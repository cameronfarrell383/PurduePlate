## Global Rules (APPLY TO EVERY TASK)

1. **Dates:** NEVER use `toISOString()` for date columns. Always format manually:
   ```typescript
   const d = new Date();
   const today = `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`;
   ```
2. **Menu date fallback:** Always use `getEffectiveMenuDate()` from `src/utils/meals.ts` when querying menu data. This returns yesterday's date if today has no scraped data yet.
3. **Colors:** All colors from ThemeContext. Approved hardcoded accents: maroon `#8B1E3F`, orange `#E87722`, blue `#5B7FFF`, yellow `#FFD60A`, green `#34C759`, red `#FF453A`.
4. **Errors:** Every Supabase call must destructure `{ error }` and handle it. Show `Alert.alert()` or toast on mutation failures.
5. **User filtering:** All queries on user-owned tables MUST filter by `user_id`.
6. **Fonts:** DM Sans (body), Outfit (headings/numbers).
7. **Git:** `git pull origin main` before starting. Push when done.
8. **No external images.** Emojis only.
9. **Keyboard:** Never create TextInput inside `.map()`, conditional renders, or functions that re-run on state change.
10. **RLS:** Every new table gets RLS policies immediately — not deferred.
11. **Console:** No `console.log` with user data. Use `console.error` for error paths only.

## Feature 1: AI Meal Planning Agent

**What:** A chat interface where students ask natural language questions about dining — "plan my meals today", "high protein lunch under 600 cal", "what should I eat at D2?" — and get personalized recommendations from today's actual VT menu data. One-tap logging from suggestions.

**Architecture:** The Claude API key NEVER touches the frontend. All AI calls go through a Supabase Edge Function that holds the key server-side. The Edge Function also enforces rate limiting (20 messages/user/day).

### Database

```sql
-- Run in Supabase SQL Editor
CREATE TABLE ai_chat_logs (
  id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role text NOT NULL CHECK (role IN ('user', 'assistant')),
  content text NOT NULL,
  meal_items jsonb DEFAULT NULL,
  created_at timestamptz DEFAULT now()
);

CREATE INDEX idx_ai_chat_user ON ai_chat_logs(user_id, created_at DESC);

-- RLS immediately
ALTER TABLE ai_chat_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can read own chats" ON ai_chat_logs FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "Users can insert own chats" ON ai_chat_logs FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "Users can delete own chats" ON ai_chat_logs FOR DELETE USING (auth.uid() = user_id);
```

```sql
-- AI context RPC — returns profile, consumed macros, and filtered menu
CREATE OR REPLACE FUNCTION get_ai_context(p_user_id uuid, p_date date)
RETURNS json AS $$
  SELECT json_build_object(
    'profile', (
      SELECT json_build_object(
        'goal_calories', goal_calories,
        'goal_protein_g', goal_protein_g,
        'goal_carbs_g', goal_carbs_g,
        'goal_fat_g', goal_fat_g,
        'dietary_needs', dietary_needs,
        'high_protein', high_protein,
        'goal', goal
      ) FROM profiles WHERE id = p_user_id
    ),
    'consumed_today', (
      SELECT json_build_object(
        'calories', COALESCE(SUM(n.calories * ml.servings), 0),
        'protein', COALESCE(SUM(n.protein_g * ml.servings), 0),
        'carbs', COALESCE(SUM(n.total_carbs_g * ml.servings), 0),
        'fat', COALESCE(SUM(n.total_fat_g * ml.servings), 0)
      )
      FROM meal_logs ml
      JOIN nutrition n ON n.menu_item_id = ml.menu_item_id
      WHERE ml.user_id = p_user_id AND ml.date = p_date
    ),
    'todays_menu', (
      SELECT json_agg(json_build_object(
        'id', mi.id,
        'hall', dh.name,
        'meal', mi.meal,
        'station', mi.station,
        'name', mi.name,
        'calories', n.calories,
        'protein_g', n.protein_g,
        'carbs_g', n.total_carbs_g,
        'fat_g', n.total_fat_g,
        'dietary_flags', mi.dietary_flags
      ))
      FROM menu_items mi
      JOIN dining_halls dh ON dh.id = mi.dining_hall_id
      JOIN nutrition n ON n.menu_item_id = mi.id
      WHERE mi.date = p_date
    )
  );
$$ LANGUAGE sql STABLE;
```

## Feature 2: Weekly Nutrition Report

**What:** A comprehensive 7-day report showing daily calorie chart, macro averages, hydration summary, weight trend, streaks, and goal adherence. Navigable by week.

### Database

```sql
-- Weekly report RPC — returns 7-day nutrition, water, weight, and streak data
CREATE OR REPLACE FUNCTION get_weekly_report(p_user_id uuid, p_end_date date)
RETURNS json AS $$
  SELECT json_build_object(
    'daily_totals', (
      SELECT COALESCE(json_agg(day_row ORDER BY day_row.log_date), '[]'::json)
      FROM (
        SELECT
          ml.date AS log_date,
          COALESCE(SUM(n.calories * ml.servings), 0) AS calories,
          COALESCE(SUM(n.protein_g * ml.servings), 0) AS protein,
          COALESCE(SUM(n.total_carbs_g * ml.servings), 0) AS carbs,
          COALESCE(SUM(n.total_fat_g * ml.servings), 0) AS fat,
          COUNT(DISTINCT ml.id) AS meals_logged
        FROM meal_logs ml
        JOIN nutrition n ON n.menu_item_id = ml.menu_item_id
        WHERE ml.user_id = p_user_id
          AND ml.date > p_end_date - 7
          AND ml.date <= p_end_date
        GROUP BY ml.date
      ) day_row
    ),
    'goals', (
      SELECT json_build_object(
        'calories', goal_calories,
        'protein', goal_protein_g,
        'carbs', goal_carbs_g,
        'fat', goal_fat_g,
        'water_goal_oz', COALESCE(water_goal_oz, 64)
      ) FROM profiles WHERE id = p_user_id
    ),
    'water_totals', (
      SELECT COALESCE(json_agg(json_build_object(
        'date', date,
        'total_oz', total_oz
      ) ORDER BY date), '[]'::json)
      FROM water_logs
      WHERE user_id = p_user_id
        AND date > p_end_date - 7
        AND date <= p_end_date
    ),
    'weight_entries', (
      SELECT COALESCE(json_agg(json_build_object(
        'date', date,
        'weight', weight
      ) ORDER BY date), '[]'::json)
      FROM weight_logs
      WHERE user_id = p_user_id
        AND date > p_end_date - 7
        AND date <= p_end_date
    ),
    'streak', (
      WITH consecutive_days AS (
        SELECT DISTINCT date
        FROM meal_logs
        WHERE user_id = p_user_id AND date <= p_end_date
        ORDER BY date DESC
      ),
      streak_calc AS (
        SELECT date, 
               date - (ROW_NUMBER() OVER (ORDER BY date DESC))::int AS grp
        FROM consecutive_days
      )
      SELECT COUNT(*)
      FROM streak_calc
      WHERE grp = (SELECT grp FROM streak_calc ORDER BY date DESC LIMIT 1)
    )
  );
$$ LANGUAGE sql STABLE;
```

**Changes from Bandhip's version:**
- `water_totals` uses `total_oz` (our actual column) instead of `glasses`
- Added `weight_entries` for weight trend chart
- Added `water_goal_oz` to goals
- Streak is a real consecutive-day calculation, not just count of distinct dates
- COALESCE wraps all json_agg calls to return `[]` instead of null when no data

## Cost & Rate Limiting Notes

Claude Sonnet API pricing (approximate):
- Input: ~$3 per million tokens
- Output: ~$15 per million tokens
- Average AI meal plan query with full context: ~4K input tokens, ~800 output tokens
- Cost per query: ~$0.024
- At 20 queries/user/day, 100 active users: ~$48/day max

Rate limits in the Edge Function:
- 20 messages per user per day (resets at midnight ET)
- Max input length: 500 characters per message
- Max history sent to Claude: last 10 messages (not full history)
- If rate limited, return friendly message: "You've used all 20 AI messages for today. They reset at midnight!"
